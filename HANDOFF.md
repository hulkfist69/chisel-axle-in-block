---

## 1. What this mod is for

VS has an "axle in block" feature: a mechanical axle (rotating shaft from the mechanical power network) can be embedded inside a solid block, so the block visually looks normal but power runs through it. VS also has a chisel system that lets players carve sub-block geometry (microblocks).

**The two systems do not currently work together.** Chiseling a block that holds an axle either destroys the axle or strips it from the network, and there is no way to drop a pre-chiseled block onto an existing axle position.

This mod's job: make those two systems coexist.

### Two specific user stories

1. **Chisel around an existing axle.**
   - Player places an axle, then puts a regular block over it (axle-in-block), then chisels that block.
   - After chiseling, the axle is still there, still spinning, still part of the same mechanical network.
   - The block's appearance is whatever the player chiseled it into.

2. **Place a chiseled block onto an axle.**
   - Player has an axle running through open space.
   - Player takes a pre-chiseled microblock from inventory and places it at the axle's position.
   - The microblock takes over the visual; the axle BE persists underneath; power keeps flowing.

Both directions should round-trip without losing axle state (orientation, network membership, rotation phase).

## 3. Architecture branch points

Based on what the discovery dump reports, the implementation forks:

### A. If axles use a BlockBehavior

Data lives on the block instance (one block class with the behavior attached). Changing the parent block to a microblock means losing the behavior. We would need to:

- Hook the chisel operation so it copies the axle behavior's state onto the new microblock.
- Add a hook on microblock placement that grafts an axle behavior onto a freshly-placed microblock if the position already had axle state.

### B. If axles use a BlockEntity (most likely)

The rotating data lives on a BE attached to the position. The parent block can change while the BE persists. Much cleaner. Likely approach:

- Hook the chisel "this block is becoming a microblock" path so the existing BE is preserved instead of destroyed.
- Hook the microblock placement path so it does not wipe an existing axle BE at the destination position.
- Possibly add a BlockBehavior on `BlockMicroBlock` that forwards "is this a power-network candidate?" queries to the axle BE.

Both routes will rely on Harmony patches discovered via reflection at runtime (same pattern used in Move Wrench — `Vintagestory.GameContent.Mechanics.*` class names varied across VS versions there too).

## 6. Lessons from Move Wrench that apply here

These were paid for in a lot of iteration; do not rediscover them.

- **VS 1.22 uses .NET 10.** `csproj` must say `net10.0`. `net7.0` produces 12 build errors from `System.Runtime` version conflicts.
- **`BlockDoor`, `BlockTrapDoor`, etc. do not declare their own `GetCollisionBoxes`-style methods in 1.22.** Patches targeting subclass `[HarmonyPatch(typeof(BlockDoor), ...)]` fail because the method is inherited from base `Block`. Patch the base and gate on type checks inside the prefix.
- **When discovery is needed, dump fields by TYPE not by NAME.** Move Wrench wasted iterations searching for "CurrentBlockSelection" by name when the actual backing field had a different name; searching for any field of type `BlockSelection` found it immediately.
- **Reflection-based patches via `RuntimePatches.cs` are how to handle classes whose names you can't be sure of.** Compile-time `[HarmonyPatch(typeof(X), ...)]` blows up at startup if `X` doesn't exist; runtime reflection lets you log and continue.
- **The user does not have VS installed on the dev machine.** Test code paths are unverifiable from the Mac side. Plan for "first compile error blocks everything" and write defensively: try/catch around reflection, never `mesh.Translate` on a null/zero-vertex mesh, etc.
- **Each user round-trip is slow** (full VS restart, paste log, fix, rebuild, copy, retest). Bias toward bundling diagnostics into the same push as the next fix rather than making them separate iterations.
